# 構造設計ルール

## 目的

生成する書類の種類・構成・強調ポイントを事前に決定することで、
採用担当者が読んだときに「ひと目で価値が伝わる」書類の骨格を設計する。

---

## 職務経歴書の標準構造

```
1. ヘッダー（氏名・連絡先・更新日）
2. 職務要約（3〜5行）
   - 現職の役割・強み・転職意図を凝縮
3. スキルサマリー
   - 言語・FW・インフラ・ツール・専門領域
4. 職務経歴（逆時系列：新しい順）
   - 各社：社名・在籍期間・役割
   - 主要プロジェクト（STAR形式）
   - 担当技術・チーム規模
5. 保有スキル・実績ハイライト
   - 横断的貢献・技術フォロー・改善実績
6. 資格・その他（任意）
```

## 自己PR文の標準構造

```
1. 導入（1〜2文）
   - 最大の強みまたは転職動機を端的に
2. 実績根拠（2〜3段落）
   - 各段落：STAR形式の具体エピソード
   - 定量値・規模感・役割を明示
3. 転職先への貢献（1〜2段落）
   - ターゲット企業・役割での活躍イメージ
   - 現職の経験との接続
4. クロージング（1〜2文）
   - 意欲・期待・一致点を簡潔に
```

---

## 各パーツの設計指針

### 職務要約
- 採用担当者が最初に読む最重要パーツ
- 「〇〇領域で△年の経験を持つシニアエンジニア。〜を強みとし、〜を目指す」のフォーマットを基本とする
- 現職のキャリアがひと言で把握できるように設計する

### 主要プロジェクト（STAR形式）
- 2〜4件を厳選して詳述する
- 数値・スケールを必ず入れる
- 「自分が何を判断・設計したか」を具体的に書く（実装だけでなく判断・リードを前景化）

### スキルサマリー
- 主力スキルと補助スキルを分けて記載する
- 「実務経験あり」と「知識レベル」を混在させない

### 横断的貢献・技術フォロー実績
- シニアエンジニア・テックリード相当の役割をアピールするために重要
- 複数チームへの関与・標準化推進・メンタリングの実績を具体的に記述する

---

## 適応的な構造選択

| 条件 | 構造の変更点 |
|------|------------|
| 転職先が完全未定 | ターゲット企業への言及を省き汎用化 |
| 職務経歴書のみ作成 | 自己PR構造は設計しない |
| ページ数制限がある | プロジェクト詳述を2件に絞る |
| 経験年数が10年以上 | 直近5年を詳述、それ以前は概略のみ |
| 求人票が提供されている | 職務要約と自己PR導入を求人要件に合わせてカスタマイズ |

---

## 出力

構造設計結果を次のステージへ引き渡す：

```markdown
## 構造設計結果（ステージ2.1）

- 作成する書類：[職務経歴書 / 自己PR / 両方]
- 採用する構造パターン：[標準 / カスタム（理由）]
- 強調するセクション：[セクション名と理由]
- 省略するセクション：[セクション名と理由]
- 訴求の核心：[ひと言で]
```
