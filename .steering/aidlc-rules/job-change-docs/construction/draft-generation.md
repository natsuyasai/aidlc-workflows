# ドラフト生成ルール

## 目的

構造設計とキャリア実績分析をもとに、採用担当者・面接官に訴求力のある
職務経歴書・自己PR文のドラフトを生成する。

---

## 執筆の基本原則

### 1. 具体性を最優先にする
- 「大規模サービスの開発経験あり」ではなく「MAU 500万のサービスでバックエンドを担当」
- 「チームの技術フォローを行った」ではなく「5チーム・30名のエンジニアに週次でアーキテクチャレビューを実施」

### 2. 主体性・判断を前景化する
- 実装だけでなく「なぜその設計を選んだか」「何を判断したか」を書く
- シニアエンジニアとしての技術的意思決定・設計責任を明示する

### 3. 数値・スケールを積極的に使う
- 直接の数値がない場合は「チーム規模 8名」「対応チーム数 3〜5」など規模感で代替する
- 改善率・削減量・達成率など変化量を数値で示す

### 4. 読み手を意識したトーンを保つ
- ビジネス文書として適切な敬語・丁寧語を使用する
- 一人称は「私」で統一する
- 専門用語は適度に使いつつ、コンテキストがなくても伝わる書き方にする

---

## フレーズ集

### 職務要約のフレーズ例
- 「〇〇領域で△年の開発経験を持つシニアエンジニア。〜を強みとし、現在は〜に従事。」
- 「バックエンド開発を軸に、複数チームへの技術支援・アーキテクチャ設計を担う。」
- 「技術的専門性と横断的なチーム支援の両立を強みとする。」

### 横断的技術支援の表現例
- 「同一製品内の複数チームに対し、設計レビュー・技術相談・コードレビュー基準の整備を担当した。」
- 「チームをまたいだ技術標準化・共通基盤の整備を主導した。」
- 「新メンバーへのオンボーディング支援および技術メンタリングを継続的に実施した。」

### 実績の書き方パターン
```
【課題】〇〇という課題が存在した。
【役割】△としてアーキテクチャ設計・技術選定を担当した。
【実施内容】□□を設計・導入し、チームへの展開を主導した。
【成果】〜が〇〇%改善し、△△が達成された。
```

## 出力形式

### 職務経歴書（Markdown形式）

```markdown
# 職務経歴書

**更新日：** YYYY年MM月

---

## 職務要約

（3〜5行）

---

## スキルサマリー

| カテゴリ | スキル・技術 |
|---------|------------|
| 言語 | |
| フレームワーク | |
| インフラ・クラウド | |
| その他ツール | |
| 専門領域 | |

---

## 職務経歴

### [会社名]（YYYY年MM月〜現在 / YYYY年MM月）

**役割：** [役職・担当領域]

#### [プロジェクト名 / 担当業務名]
- **期間：** YYYY年MM月〜YYYY年MM月
- **チーム規模：** 〇名
- **技術スタック：** [使用技術]
- **担当内容：** [STAR形式で記述]
- **成果：** [定量値または定性的インパクト]

---

## 実績ハイライト

- [横断的貢献・技術フォロー・改善実績など]

---

## 資格・その他（任意）

- [資格・登壇・執筆など]
```

### 自己PR文（Markdown形式）

```markdown
## 自己PR

（導入：1〜2文）

（実績根拠1：STAR形式エピソード）

（実績根拠2：STAR形式エピソード）

（転職先への貢献：1〜2段落）

（クロージング：1〜2文）
```
