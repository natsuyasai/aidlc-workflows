# レビュー・洗練ルール

## 目的

ユーザーに提示する前に生成したドラフトを要件・実績・訴求力の観点で自己レビューし、
フィードバックに基づく反復的な改善をガイドする。

---

## 提示前チェックリスト

### 内容の正確性・完全性
- [ ] 要件ブリーフに記載されたすべての主要実績が書類に反映されている
- [ ] 職種・役割・在籍期間の記述が矛盾なく一貫している
- [ ] 技術スタックの記述が実際の経験レベルと対応している
- [ ] 定量値・数値が提供された情報と整合している

### 訴求力・差別化
- [ ] 職務要約を30秒で読んだ場合に「どんな人か」が即座に伝わる
- [ ] シニアエンジニアとしての判断・設計責任が具体的に書かれている
- [ ] 横断的技術支援・複数チーム関与の実績が適切に前景化されている
- [ ] 実装だけでなく「なぜ」「何を判断したか」が記述されている

### 構成・フォーマット
- [ ] 採用した構造設計に従って書類が構成されている
- [ ] 各セクションの分量バランスが適切である（要約が長すぎない、詳述が短すぎない）
- [ ] Markdown 形式が崩れていない
- [ ] 敬語・文体・一人称が統一されている

### ターゲット適合性（ターゲット分析を実施した場合）
- [ ] 訴求メッセージが求人要件の必須スキルに対応している
- [ ] 自己PRの結論部がターゲット企業・役割への貢献に言及している
- [ ] 差分・ギャップがある場合に適切な補完表現が入っている

---

## 洗練のガイダンス

ユーザーから修正依頼を受けた場合：

| フィードバックの種類 | 対応方法 |
|---------------------|----------|
| 「もっと具体的に書いてほしい」 | STAR形式の各要素を展開し、数値・規模感を追加する |
| 「この実績を強調してほしい」 | 対象実績を上位に移動・詳述し、職務要約にも反映する |
| 「特定企業向けにカスタマイズしたい」 | ステージ1.4（ターゲット分析）を実行し、求人要件に合わせて書き直す |
| 「もっと簡潔にしてほしい」 | 詳述プロジェクトを絞り、箇条書きを短縮する |
| 「謙虚すぎる / 大げさに感じる」 | トーン調整：事実ベースの記述を保ちながら表現の強弱を調整する |
| 「自己PRの流れがつかみにくい」 | 段落構成を見直し、論理の流れ（課題→実績→貢献）を強化する |

---

## ユーザーへの提示方法

```
職務経歴書（または自己PR）のドラフトを作成しました。

**ご確認いただきたいポイント：**
1. [重要確認項目1]
2. [重要確認項目2]

修正・追加・変更があればお知らせください。
特定企業向けへのカスタマイズも対応可能です。
```

数値や固有名詞は「提供された情報をもとに記載しています。実際の値と異なる場合は修正をお知らせください」と明示する。
