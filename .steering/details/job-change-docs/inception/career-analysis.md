# キャリア実績分析ルール

## 目的

ユーザーが提供したキャリア情報を整理・深堀りし、
職務経歴書・自己PRで最大限に訴求できるエピソード・実績・強みを抽出する。

---

## 分析の観点

### 1. 実績の定量化

数値・スケールで表現できる要素を探す：

| 観点 | 確認する内容 |
|------|-------------|
| **規模感** | ユーザー数・DAU・トランザクション数・データ量・チーム人数 |
| **改善効果** | レスポンスタイム削減・コスト削減率・障害件数の変化 |
| **開発生産性** | リリース頻度・デプロイ時間・テストカバレッジの変化 |
| **ビジネス貢献** | 売上への貢献・新機能のKPI達成・顧客満足度 |

定量化できない場合は「定性的インパクト」（例：チーム全体の技術レベル向上、横断的な標準化推進）として記録する。

### 2. 役割・貢献の明確化

技術的実装だけでなく、以下の観点でも深堀りする：

- **技術リード的貢献**：設計判断・アーキテクチャ提案・技術選定
- **チーム貢献**：コードレビュー文化の醸成・メンタリング・オンボーディング
- **横断的貢献**：複数チームへの技術フォロー・標準化・共通基盤の整備
- **プロセス改善**：開発フロー・CI/CD・テスト体制の改善

### 3. 強みの分類

抽出した実績を以下の4カテゴリに分類する：

| カテゴリ | 内容 |
|---------|------|
| **技術的専門性** | 特定技術領域での深い知識・経験 |
| **技術横断・組織貢献** | 複数チーム・レイヤーにまたがる貢献 |
| **問題解決・改善** | 課題発見から解決まで主導した経験 |
| **コミュニケーション・リード** | 技術的意思決定や他職種との協力 |

### 4. ストーリー構造の設計

各実績をSTAR形式で構造化する：

```
Situation（状況）：どのような課題・背景があったか
Task（役割）     ：自分の担当・責任範囲は何だったか
Action（行動）   ：何を考え、どう動いたか（技術的判断を含む）
Result（結果）   ：何が変わったか・達成したか（できれば定量値）
```
