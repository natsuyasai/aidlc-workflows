# コンテキスト検出ルール

## 目的

ユーザーが提供した情報をもとに、新しいワークフロー定義に必要な情報がどの程度揃っているかを評価し、
続けて実行すべきインセプションステージを決定する。

---

## 検出チェックリスト

ユーザーの入力を以下の5つの次元で評価する：

| 次元 | 確認内容 | 状態 |
|------|----------|------|
| **作業名・目的** | 対象となる作業の名称と「何を達成するか」が明確か | [ ] 判明 / [ ] 不明 |
| **ライフサイクル** | 作業の開始〜完了までの自然な流れが判明しているか | [ ] 判明 / [ ] 不明 |
| **インプット** | AIに与えるべき情報・コンテキスト（作業を始めるために必要な材料）が判明しているか | [ ] 判明 / [ ] 不明 |
| **アウトプット** | 作業の成果物・完了条件が明確か | [ ] 判明 / [ ] 不明 |
| **繰り返し構造** | この作業が定期的・反復的に発生するものか、一回限りのものかが判明しているか | [ ] 判明 / [ ] 不明 |

---

## 判断ロジック

```
4〜5/5 次元が判明している場合：
    → ステージ1.3（作業特性分析）から開始

2〜3/5 次元が判明している場合：
    → 不明な次元のみを対象にステージ1.2（要件分析）を実行

0〜1/5 次元が判明している場合：
    → インセプション全体を実行（ステージ1.2 → 1.3 → 1.4）
```

---

## 類似ワークフローの照合

検出時に、既存ワークフローとの類似度も評価する：

| 既存ワークフロー | 類似する作業の特徴 |
|----------------|-----------------|
| `business-email` | テキスト成果物・受信者が存在・トーン調整が必要 |
| `tech-blog` | 長文テキスト成果物・構成設計が重要・公開を伴う |

類似ワークフローが存在する場合：
- 参照元として明示し、差分のみを新規設計することを提案する
- ユーザーの承認を得てから設計を進める

---

## 出力

検出結果をセッション状態に記録する：

```markdown
## コンテキスト検出結果
- 作業名・目的：[判明 / 不明]
- ライフサイクル：[判明 / 不明]
- インプット：[判明 / 不明]
- アウトプット：[判明 / 不明]
- 繰り返し構造：[判明 / 不明]
- 類似ワークフロー：[なし / business-email / tech-blog]
- 判断：[1.3から開始 / 部分的な要件収集 / インセプション全体実行]
```
