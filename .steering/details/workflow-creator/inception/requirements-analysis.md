# 要件分析ルール

## 目的

コンテキスト検出で「不明」と判定された次元について、構造化された質問で
新しいワークフロー定義に必要な情報を収集する。

---

## 質問バンク

コンテキスト検出で「不明」とされた次元についてのみ質問する。一度に質問するのは最大5問まで。

### 作業名・目的に関する質問
- 新しいワークフローで支援したい作業の名前を教えてください
- この作業で最終的に何を達成しますか？（例：文書が完成する・意思決定ができる・設定が整う）
- この作業を AI に支援させようと思ったきっかけや背景はありますか？

### ライフサイクルに関する質問
- この作業は「始め → 中盤 → 仕上げ」のような段階に自然に分かれますか？
  それぞれの段階では何をしていますか？
- 作業の途中で「ここで一度確認・承認が必要」というポイントはありますか？
- 作業が完了したと判断できる条件・成果物は何ですか？

### インプットに関する質問
- この作業を始めるために AI に渡す情報は何ですか？
  （例：テーマ・背景情報・制約・過去の類似事例・テンプレートなど）
- 毎回変わる情報と、毎回共通して使う情報に分けると、どうなりますか？
- 情報が不足している場合、AI はどのように補えばよいですか？
  （質問して収集する / デフォルト値を使う / スキップする）

### アウトプットに関する質問
- 作業の成果物はどのような形式ですか？
  （文書 / 箇条書きリスト / コード / 表 / 図 / その他）
- 成果物の品質を判断する基準は何ですか？
  （「〇〇が含まれていること」「〇〇の形式に従っていること」など）
- 成果物はその後どのように使われますか？（誰かに渡す / 保存する / 即実行する）

### 繰り返し構造に関する質問
- この作業はどのくらいの頻度で発生しますか？
  （毎日 / 週次 / 月次 / 半期 / 不定期 / 特定イベント時のみ）
- 毎回まったく同じ手順ですか？それとも状況によって手順が変わりますか？
- 過去の作業結果を次回に活かすような「蓄積・参照」は必要ですか？

---

## 出力形式

回答を収集したら、**ワークフロー要件ブリーフ**としてまとめる：

```markdown
## ワークフロー要件ブリーフ

- **作業名**：[作業の名称]
- **目的**：[この作業で達成すること]
- **発生頻度**：[頻度・タイミング]
- **インプット**：
  - 毎回変わる情報：[一覧]
  - 毎回共通の情報：[一覧]
- **アウトプット**：[成果物の形式・内容]
- **完了条件**：[何ができれば作業完了か]
- **品質基準**：[成果物の品質を判断する基準]
- **後工程**：[成果物の使われ方・渡し先]
- **特記事項**：[制約・注意点・背景情報]
```

この要件ブリーフを作業特性分析（ステージ1.3）への正式なインプットとして引き渡す。
