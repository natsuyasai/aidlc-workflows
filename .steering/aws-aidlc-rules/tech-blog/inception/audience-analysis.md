# 読者・目的分析ルール

## 目的

読者のスキルレベルと目的を詳細に把握し、記事全体のトーン・説明の深さ・コード量を決定する。

---

## 読者プロファイル

以下の観点で読者像を整理する：

| 要素 | 確認内容 |
|------|----------|
| **スキルレベル** | 入門 / 中級 / 上級 |
| **前提知識** | どの技術・概念をすでに理解していると想定するか |
| **読む動機** | 問題解決 / 学習 / 技術トレンドのキャッチアップ / 意思決定の参考 |
| **読む環境** | 業務中のリファレンス / 通勤中のインプット / 休日の深掘り学習 |
| **期待する成果** | 手を動かして動くものを作る / 概念を理解する / 判断基準を得る |

---

## スキルレベル別の執筆指針

### 入門者向け
- 専門用語は初出時に必ず定義する
- 「なぜこれが必要か」の動機付けを省略しない
- コード例は最小構成から始め、段階的に拡張する
- エラーへの対処法・躓きやすいポイントを明示する
- 図や概念図を積極的に使う

### 中級者向け
- 基本的な概念の説明は簡潔にまとめ、深い部分に紙幅を使う
- 「なぜこの実装方法を選んだか」のトレードオフを説明する
- 実務で使えるレベルのコード例を示す
- 参考資料・公式ドキュメントへのリンクを添える

### 上級者向け
- 内部実装・パフォーマンス特性・エッジケースに踏み込む
- 設計判断の背景・制約・代替案を論じる
- コードは詳細なコメントより設計の意図の説明を重視する
- ベンチマーク・計測結果など定量データを活用する

---

## 記事の目的分類

記事が達成すべき主目的を明確にする：

| 目的 | 特徴 | 重視する指標 |
|------|------|------------|
| **教育**（How-to） | 読者が手順を再現できる | 手順の完全性・再現性 |
| **啓発**（Why-to） | 読者の考え方・認識を変える | 論拠の質・説得力 |
| **情報提供**（What-is） | 読者が概念を理解できる | 正確性・わかりやすさ |
| **意思決定支援** | 読者が選択肢を比較できる | 公平性・比較軸の明確さ |
| **体験共有** | 読者が追体験・共感できる | リアリティ・具体性 |

---

## アンチパターン（避けるべきこと）

| アンチパターン | 対策 |
|--------------|------|
| 読者のスキルを誤って設定する | 要件ブリーフの読者層を再確認する |
| 「当然わかるはず」で説明を省く | 入門者フレンドリーな補足注釈を入れる |
| コードをただ貼るだけで説明しない | コードの前後に「何をするか」「なぜこう書くか」を説明する |
| 読者に何も期待しない（CTA不在） | 最後に次のステップを1つ提示する |

---

## 出力

要件ブリーフに以下を追記する：

```markdown
- **読者プロファイル**：[スキルレベル・動機・期待成果]
- **執筆深度**：[入門 / 中級 / 上級向けの指針]
- **記事の主目的**：[教育 / 啓発 / 情報提供 / 意思決定支援 / 体験共有]
- **注意点**：[特定の読者層への配慮事項]
```
