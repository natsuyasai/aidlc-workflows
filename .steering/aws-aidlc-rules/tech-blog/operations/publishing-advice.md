# 公開アドバイスルール

## 目的

ユーザーが求めた場合に、公開プラットフォームの選定・タグ設定・拡散戦略・フォローアップについて推奨事項を提供する。

---

## プラットフォーム選定ガイド

| プラットフォーム | 特徴 | 向いている記事 |
|----------------|------|--------------|
| **Zenn** | エンジニア特化、Markdown品質が高い、スクラップ機能あり | 技術解説・チュートリアル・シリーズ記事 |
| **Qiita** | 国内エンジニア最大手、検索流入が多い | 入門〜中級向けHow-to・ニッチな技術Tip |
| **note** | 文章重視、エンジニア以外にも届く | 体験談・キャリア・チームのカルチャー |
| **Medium / Dev.to** | 英語圏リーチ | 英語記事・海外読者を想定した技術解説 |
| **社内ブログ / Confluence** | 社内ナレッジ蓄積 | 業務固有の知見・障害報告・意思決定記録 |
| **個人ブログ（GitHub Pages / Hugo など）** | 完全なコントロール、ポートフォリオ兼用 | 長期的な発信・ブランディング |

**複数プラットフォームへの転載**：初出から1〜2週間後に転載するのが一般的。転載元を明記し、canonical URLを設定する。

---

## タグ・カテゴリの設定

効果的なタグ選定のための指針：

### タグの種類と使い方
| タグの種類 | 例 | 効果 |
|-----------|-----|------|
| 技術名（固有名詞） | Go、Kubernetes、React | 技術ユーザーへの直接リーチ |
| 記事の種類 | チュートリアル、入門、tips | 検索意図と合致 |
| 問題領域 | パフォーマンス、セキュリティ、CI/CD | 課題を持つ読者へのリーチ |
| 対象者 | 初心者向け、バックエンド | 読者セグメントの絞り込み |

### タグ設定の原則
- 具体的すぎるタグ（検索ボリュームがほぼない）は避ける
- 記事の内容と無関係なタグを付けない
- Zenn・Qiitaは5タグ以内を推奨

---

## 公開タイミング

| 条件 | 推奨タイミング |
|------|--------------|
| 一般的な技術記事 | 火〜木曜日の9:00〜12:00（エンジニアの情報収集時間帯） |
| トレンド便乗記事 | リリース・発表から48時間以内（鮮度が命） |
| 入門系・週末学習向け | 金曜日の夕方〜土曜日の午前 |
| アドベントカレンダー | 担当日の0:00〜9:00の間 |

## 拡散戦略

### 公開直後のアクション
1. X（旧Twitter）にポスト：タイトル + 記事URL + 1〜2行のサマリー + 関連ハッシュタグ
2. 社内Slackの技術共有チャンネルに投稿
3. 関連するOSS・コミュニティのDiscord/Slackがあれば共有
4. LinkedIn（英語記事・キャリア関連の場合）

### X（旧Twitter）投稿の型
```
[記事タイトル]

[1〜2行のサマリー：読者が得られるもの]

👇 記事はこちら
[URL]

#タグ1 #タグ2
```

### 継続的な拡散
- 内容が陳腐化していない記事は3〜6ヶ月後に再投稿
- シリーズ記事の次回公開時に前回記事を改めて紹介

---

## 公開後のフォローアップ

| アクション | タイミング | 内容 |
|-----------|-----------|------|
| コメント返信 | 公開後48時間以内 | 質問・指摘には誠実に返答する |
| 誤り修正 | 気づき次第すぐ | 修正した旨を記事内に明記（「YYYY/MM/DD 修正」） |
| 関連記事リンク追加 | 続編公開時 | 前後の記事を相互リンクする |
| アクセス解析確認 | 公開1週間後 | 流入経路・離脱ポイントを次回の改善に活かす |

---

## 出力

ユーザーが求めた場合のみ、ドラフトの後に簡潔な箇条書きで公開アドバイスを提供する：

```
【公開アドバイス】
- 推奨プラットフォーム：Zenn（技術解説記事のため）
- 推奨タグ：Go、パフォーマンス、チュートリアル
- 推奨公開日時：火〜木曜日の午前中
- X投稿文案：[サンプルを提示]
```
