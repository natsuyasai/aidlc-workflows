# ドラフト生成ルール

## 目的

要件ブリーフ・記事構成計画・構造設計をもとに、技術ブログ記事の完全なドラフトを作成する。

---

## 執筆の基本原則

### 1. 一貫した人称・トーンを保つ
記事全体で話者のスタンスを統一する：
- **ですます調**：入門〜中級者向け、親しみやすさ重視
- **である調**：上級者向け・論考系、簡潔さ重視
- 途中で混在させない

### 2. 1段落1メッセージ
各段落は1つの主張だけを扱う。段落が長くなる場合は分割する目安は150字。

### 3. 技術用語の扱い
- 初出時に日本語訳または説明を付ける（例：「デプロイ（本番環境への反映）」）
- 英語の固有名詞はそのまま使用（例：Kubernetes、gRPC）
- 読者レベルに応じて「既知として扱う」用語の範囲を調整する

### 4. コードの説明は前後に書く
コードブロックの前：「このコードは〇〇を行います」
コードブロックの後：「〇行目で〜しているのがポイントです」

### 5. 主観と客観を使い分ける
- 体験談・意見：主観を明示する（「私の場合は〜」「〜だと感じています」）
- 技術解説・比較：客観的な根拠を示す（ベンチマーク結果・公式ドキュメントの引用）

---

## 文体・表現のガイドライン

### 読みやすい文体のために
| 避けるべき表現 | 改善例 |
|-------------|--------|
| 〜であると考えられる | 〜です（断定できる場合） / 〜と思います（主観の場合） |
| 〜することができます | 〜できます |
| 非常に重要な | 重要な（「非常に」は多用すると効果が薄れる） |
| 〜等 | 具体的に列挙するか「〜など」に統一 |
| 上記の通り | 参照先を明示する（「前のセクションで示した〜」） |

### 技術記事特有の表現
| 場面 | 推奨表現 |
|------|---------|
| 手順を示す | 「〜します」「〜してください」（体言止めより動詞で終わらせる） |
| 注意を促す | 「⚠️ 注意：〜」または「> **注意**：〜」（引用ブロックで強調） |
| 補足説明 | 「> **補足**：〜」または脚注形式 |
| コマンドの説明 | 「以下のコマンドを実行します：」の後にコードブロック |
| 結果の確認 | 「〜が表示されれば成功です」 |

## コードブロックの書き方

### 基本フォーマット
````markdown
```言語名
// コード
```
````

### ファイル名を示す場合
````markdown
```go:main.go
package main
// ...
```
````

### 実行コマンドと出力を示す場合
````markdown
```bash
$ go run main.go
```

```
Hello, World!   ← 出力例
```
````

### コードの一部を省略する場合
```go
func main() {
    // 前の処理（省略）

    result := processData(input)  // ← ここが重要

    // 後の処理（省略）
}
```

## よく使う表現集（日本語技術ブログ）

### 記事の冒頭
- `この記事では〜について解説します。`
- `〜を使って〇〇を実現する方法を紹介します。`
- `〜に詰まったので、解決策をまとめました。`
- `〜と〜を比較し、それぞれの使いどころを整理しました。`

### セクション間のつなぎ
- `ここまでで〜の基本を理解しました。次は〜を見ていきます。`
- `〜の仕組みが分かったところで、実際に実装してみましょう。`
- `基本的な使い方の次は、実務でよく使うパターンを紹介します。`

### まとめ
- `この記事では以下のことを解説しました：`
- `〜を使うことで、〇〇という課題を解決できます。`
- `次のステップとして、〜を試してみることをおすすめします。`

## 出力形式

記事全文をMarkdown形式で提示する：

```markdown
# [タイトル]

## はじめに
[リード文]

## [セクション1]
[本文]

```言語
[コード例]
```

## まとめ
[要点・CTA・参考リンク]
```

記事の冒頭に以下のメタ情報を付ける：

```
---
想定読者：[スキルレベル]
記事種別：[種別]
推定読了時間：[〇分]（文字数をもとに算出）
---
```
