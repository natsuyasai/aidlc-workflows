# レビュー・洗練ルール

## 目的

ユーザーに提示する前に生成したドラフトを多角的にレビューし、技術的正確性・読みやすさ・構成品質を確認する。

---

## 提示前チェックリスト

すべてのドラフトを提示する前に以下を確認する：

### 技術的正確性
- [ ] コードが構文的に正しい（言語ごとのルールに違反していない）
- [ ] コマンドや手順が再現可能な順序になっている
- [ ] バージョン・固有名詞・APIの名称が正確に記載されている
- [ ] 誤った断言・過度な一般化がない（「〜は常に〜」「〜は絶対に〜」など）
- [ ] 古い情報・非推奨の手法を使っていないか注記がある

### 読みやすさ
- [ ] リード文で記事の内容・対象読者・得られるものが明確
- [ ] 1セクション1トピックの原則が守られている
- [ ] 段落が長すぎない（目安：6〜8行以内）
- [ ] 技術用語は初出時に説明されている
- [ ] コードの前後に説明がある

### 構成
- [ ] タイトルが内容を正確に表している
- [ ] 見出しを見ただけで記事の流れがわかる
- [ ] まとめセクションに要点とCTAがある
- [ ] セクション間の繋ぎが自然である

### 読者適合性
- [ ] 説明の深さが想定読者のスキルレベルに合っている
- [ ] 入門者向けの場合：「なぜ」の説明が省略されていない
- [ ] 上級者向けの場合：自明な説明で紙幅を使いすぎていない

### フォーマット（Markdown）
- [ ] コードブロックに言語識別子がついている
- [ ] 見出しレベルが適切（h1はタイトルのみ、本文はh2〜h3）
- [ ] リストは並列な項目にのみ使っている（並列でない場合は文章で書く）

---

## 洗練のガイダンス

ユーザーから修正依頼を受けた場合：

| フィードバックの種類 | 対応方法 |
|---------------------|----------|
| 「難しすぎる」 | 専門用語に説明を追加、前提知識の説明を追加、例示を増やす |
| 「簡単すぎる・物足りない」 | 基礎説明を圧縮、内部実装・トレードオフ・発展的内容を追加 |
| 「長すぎる」 | 各セクションの要点以外を削除、冗長な言い回しを短縮 |
| 「コードを増やしたい」 | 動作する完全な例・実用的な拡張例を追加 |
| 「もっと自分の言葉で書きたい」 | 体験談・一人称表現・具体的なエピソードを挿入するよう促す |
| 「全体的に書き直したい」 | 更新された要件でドラフト生成に戻る |

---

## ユーザーへの提示方法

チェックリストを通過したら、以下の形式で提示する：

1. **ドラフト本体** — Markdown全文
2. **レビューメモ** — 前提・懸念点・要確認事項（最大3点）
3. **修正の案内** — フィードバックを促す一言

例：

```
以下のドラフトを作成しました。

---
[ドラフト全文]
---

【レビューメモ】
- コード例はGo 1.22を想定しています。バージョンが異なる場合はお知らせください
- 「前提知識：Dockerの基本操作」を冒頭に追加しました
- まとめのCTAは「公式チュートリアルを試す」としましたが、変更可能です

修正希望があればお知らせください。
```
