# 作業特性分析ルール

## 目的

要件ブリーフをもとに対象作業の本質的な構造を分析し、
AI-DLC の3フェーズ（インセプション・コンストラクション・オペレーション）に
どのようにマッピングするかを決定する。

---

## 分析ステップ

### ステップ1：作業の構造パターンを特定する

対象作業が以下のどのパターンに最も近いかを判定する：

| パターン | 特徴 | 代表例 |
|---------|------|--------|
| **生成型** | ゼロから成果物を作り出す | 文書作成・コード生成・企画立案 |
| **変換型** | 既存の情報を別の形式・視点に変換する | 要約・翻訳・リファクタリング・分析 |
| **評価型** | 対象を基準に照らして判断・採点する | コードレビュー・目標評価・候補選定 |
| **計画型** | 将来の行動・リソース・スケジュールを設計する | プロジェクト計画・目標設定・ロードマップ作成 |
| **対話型** | 複数回のやりとりを通じて成果物を育てる | インタビュー・ブレインストーミング・要件定義 |

複数のパターンが混在する場合はすべて記録する。

---

### ステップ2：3フェーズへのマッピング

各フェーズに何が入るかを決定する：

#### インセプション（何を・なぜやるかを明確にする）
- **コンテキスト検出の次元**：作業開始に必要な情報の軸は何か？
  - 例（ビジネスメール）：目的・受信者・要点・トーン・制約
  - 例（技術ブログ）：テーマ・記事種別・読者層・要点・コード・制約
- **スキップ可能な条件**：情報が十分揃っている場合にスキップできるステージはあるか？
- **必須質問の特定**：常に聞かなければならない最低限の質問は何か？

#### コンストラクション（どのように作るかを決める）
- **成果物の構造**：成果物はどのような要素から構成されるか？
- **生成の順序**：要素を生成する自然な順序はあるか？
- **品質チェックポイント**：ユーザーに提示する前に確認すべき項目は何か？
- **反復の単位**：修正依頼を受けた場合、何の単位で作り直すか？

#### オペレーション（いつ・どのように使うかを決める）
- **後工程の有無**：成果物を使う・渡す・適用するなどの後工程があるか？
- **タイミングの制約**：実行・提出・公開のベストタイミングはあるか？
- **フォローアップ**：作業完了後に追跡・確認すべきことはあるか？

---

### ステップ3：適応的実行の設計

新しいワークフローで「スキップ可能なステージ」と「常に実行するステージ」を整理する：

```markdown
## 適応的実行マップ

### 常に実行するステージ
- [ステージ名]：[常に必要な理由]

### 条件付き実行ステージ
- [ステージ名]：[実行する条件] / [スキップする条件]
```

---

### ステップ4：コンテキスト検出次元の設計

新しいワークフローのインセプションで評価すべき「情報の軸」を定義する：

```markdown
## コンテキスト検出次元

| 次元名 | 確認する内容 | 情報が揃っている基準 |
|--------|------------|---------------------|
| [次元1] | [何を確認するか] | [どうなれば「判明」とするか] |
| [次元2] | ... | ... |
```

3〜7次元が適切。多すぎると検出が複雑になる。

---

## 出力

作業特性分析の結果を以下の形式でまとめる：

```markdown
## 作業特性分析結果

- **構造パターン**：[生成型 / 変換型 / 評価型 / 計画型 / 対話型（複数可）]
- **フェーズマッピング**：
  - インセプション：[何を明確にするフェーズか]
  - コンストラクション：[何を生成・処理するフェーズか]
  - オペレーション：[後工程の内容 / 不要な場合は「なし」]
- **コンテキスト検出次元**：[次元一覧]
- **常に実行するステージ**：[一覧]
- **条件付き実行ステージ**：[条件と共に一覧]
- **設計上の注意点**：[この作業特有の考慮事項]
```

この分析結果をファイル構成計画（ステージ1.4）へ引き渡す。
